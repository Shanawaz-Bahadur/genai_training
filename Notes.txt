Ans) 1. Data Extraction 
     2. API Building (getting response in JSON format makes it easy to integrate it anywhere)
     3. Agents (Agent -> prompts -> (It inputs the Natural language text)LLM(It responds with the structured output) -> Tools (It expects the structure input))

LLM can generate 2 types of output - 1. structured output(Func - with_structured_output()) and 2. Unstructured output(Func - Output parser[To make it structured])

    i. with_structured_output() (before using this we need to define the data_format of the output)
        a. TypedDict (It defined which keys will be used with there data types with other attributes such as Annotated, Optional, Literal, but the drawback of typedDict is it doesn't give any error if the data types is not followed, so to bypass this drawback we use Pydantic)
        (When to use: Only need type hint, Don't need Validation)

        b. Pydantic (It is a Data Validation and Data parsing library for Python, it ensure that the data you work with is correct, structured and type-safe)
        (When to use: Data Validation, Default Values, Automatic type conversion)

        c. json_schema (As JSON is an universal Data format, if we are using JS in frontend and Python in BE then we can make use of json_schema to make it compatible in both cases)
        (When to use: A Standard JSON format, Don't want to import Python libraries(Pydantic))

    ii. Output Parser - In Langchain it helps convert raw LLM response into structured formats like JSON, CSV, Pydantic models and more. They ensure consistency, Validation, and ease of use in application

        a.StrOutputParser() - It is used to parse the output of a LLM and return it as a plain string

            Response from LLM -> StrOutputParser -> Return only the String(to avoid making use of result.content and it is used to avoid metadata)

        b. JSONOutputParser() - It is used to parse the output of a LLM and return it as a JSON, but without any structured output schema, thats why we move on to the next parser - StructuredOutputParser()

        c. StructuredOutputParser() - It is an output parser in Langchain that helps extract structured JSON data from LLM responses based on predefined field schema, but in this we cannot apply the validation thats why we move to the next parser - PydanticOutputParser()

==================================================> RAG <===================================================================================

    RAG is a technique that combines information retrieval(From the knowledge base) with language generation, where a models retrives relevant documents from the knowledge base and then uses them as a context to generate accurate and grounded response. 

    Benefits of RAG:

        1. Use of up-to-date response
        2. Better Privacy
        3. No limit of document size

    RAG has 4 main Components:

        1. Document Loader
        2. Text Splitters
        3. Vector Databases
        4. Retrivers

Video 12 => Document Loader

Document loaders are components in LangChain used to load data from various sources into standardized format (usually as Document objects), which can then be used for chunking, embedding, retrieval, and generation

As Data can be extracted from any of the sources like - PDF, Database, S3 Bucket, text files, thats the we need to convert it into a standardized formate which is Document object which mainly consist of  page_content and metadata

Types of Document Loader

    1. Text Loader
    2. PyPDFLoader
    3. WebBaseLoader
    4. CSVLoader

    1. Text Loader: It is a simple and commonly used document loader in LangChain that reads plain text (.txt) files and converts them into LangChain Document objects, it TextLoader() returns a list of Document objects

        Use Case

            - Ideal for loading chat logs, scraped text, transcripts, code snippets, or any plain text data into a LangChain pipeline.

        Limitation

            - Works only with .txt files    

    2. PyPDFLoader: It works page by page, if we provide 25 pages pdf it will return a list of 25 Documents which mainly consist of  page_content and metadata, it is a document loader in LangChain used to load content from PDF files and convert each page into a Document object, it also 

        Limitations:

            - It uses the PyPDF library under the hood - not great with scanned PDFs or complex layouts.

        Below are some other loader as per the usecase

            Simple, clean PDFs - PyPDFLoader
        
            PDFs with tables/columns - PDFPlumberLoader

            Scanned/image PDFs - UnstructuredPDFLoader Or AmazonTextractPOFLoader

            Need layout and image data - PyHluPDFLoader

            Want best structure extraction - UnstructuredPDFLoader

    3. WebBaseLoader: It is a document loader in LangChain used to load and extract text content from web pages (URLs).

        It uses Request(to get call the API) BeautifulSoup under the hood to parse HTML and extract visible text.

        When to Use:
            . For blogs, news articles, or public websites where the content is primarily text-based and static.

        Limitations:
            . Doesn't handle JavaScript-heavy pages well (use SeleniumURLLoader for that).
            . Loads only static content (what's in the HTML, not what loads after the page renders).

Video 13 ==> Text Splitters: It is the process of breaking large chunks of text (like articles, PDFs, HTML pages, or books) into smaller, manageable pieces (chunks) that an LLM can handle effectively.

                                        (Large Text -> Smaller chunks)

    Q) Why do we require Text Splitters
    Ans)
    1. Overcoming model limitations: Many embedding models and language models have maximum input size constraints. Splitting allows us to process documents that would otherwise exceed these limits.  
    
    2. Downstream tasks - Text Splitting improves nearly every LLM powered task  

           Task                  Why Splitting Helps
        Embedding - Short chunks yield more accurate vectors
        Semantic Search - Search results point to focused info, not noise
        Summarization - Prevents hallucination and topic drift  
    
    3. Optimizing computational resources: Working with smaller chunks of text can be more memory-efficient and allow for better parallelization of processing tasks.

    => Types of Text Splitters

        1. Length based: CharacterTextSplitter class: 

        This splitter uses a fixed-length, length-based approach. It simply splits text into chunks based on a set number of characters (or tokens if using from_tiktoken_encoder), regardless of semantic boundaries. It Ensure consistent chunk sizes. Used when Uniform text without strong structural cues, or when exact size control is critical.

        2. Text Structure based: RecusiveCharacterTextSplitter class:

        This splitter uses a hierarchical, structure-aware approach. It starts with larger natural text units (e.g., paragraphs split by \n\n), and if a chunk exceeds the chunk_size, it recursively moves to smaller units (\n, then spaces, then characters) until the chunk fits.It  Preserve semantic coherence by keeping natural boundaries intact. It is used when Texts with meaningful structure (articles, essays, code).

        3. Document Structure based: It uses RecusiveCharacterTextSplitter but with some different text_splitters which is mainly used for markdown files or any code file(Python, JS, Java) and we need to specify the language

        4. Semantic Meaning based: SemanticChunker (Its under experimentation)

    Q) Why Vector store?
    Ans) It solves the below 3 challenges faced while building the RAG application:
            1. Generating Vector Store
            2. Storage

    Q) What is Vector Store?
    Ans) It is a system designed to store and retrieve data represented as numerical  
    
    Key Features  
    
        1. Storage - Ensures that vectors and their associated metadata are retained, whether in- memory(RAM) for quick lookups or on-disk for durability and Targe-scale use.  
        
        2. Similarity Search - Helps retrieve the vectors most similar to a query vector.  (comparing query vector with all the vectors present in the vector DB)
        
        3. Indexing - Provide a data structure or method that enables fast similarity searches on high-dimensional vectors (e.g., Clustering approximate nearest neighbor lookups). Indexing is mainly used to speed-up the process of searching
        
        4. CRUD Operations - Manage the lifecycle of data-adding new vectors, reading them, updating existing entries, removing outdated vectors.  
    
    Use-cases:

        1. Semantic Search 
        2. RAG 
        3. Recommender Systems 
        4. Image/Multimedia search

    In similarity_search, k stands for the no. of Documents to be retured after searching

Video 15 ==> Retrievers 

    Q) What are Retrievers?
    Ans) A retriever is a component in LangChain that fetches relevant documents from a data source in response to a user's query.  There are multiple types of retrievers  All retrievers in LangChain are runnables

                                    Query -> Retriever-> Data Source -> Returns multiple Documents

    Types of retrievers

        1. Wikipedia Retrivers (Query -> Wikipedia retrievers API call -> Returns the Document of relevant content)

        2. Vector Store Retriever***: A Vector Store Retriever in LangChain is the most common type of retriever that lets you search and fetch documents from a vector store based on semantic similarity using vector embeddings.

        - How It Works 
        
            1. You store your documents in a vector store (like FAISS, Chroma, Weaviate) 
            2. Each document is converted into a dense vector using an embedding model 
            3. When the user enters a query:  
                . It's also turned into a vector  
                . The retriever compares the query vector with the stored vectors  
                . It retrieves the top-k most similar ones

        3. Maximum Marginal Relevance(MMR)

            MMR is an information retrieval algorithm designed to reduce redundancy in the retrieved results while maintaining high relevance to the query.

        4. Multi Query Retrivers

            Sometimes a single query might not capture all the ways information is phrased in your documents.

        5. Contextual Compression Retriever

            The Contextual Compression Retriever in LangChain is an advanced retriever that improves retrieval quality by compressing documents after retrieval - keeping only the relevant content based on the user's query.

    Imp note: It is used to improves the response of the LLM so that the most relavant context is extracted from the vectore store.

Video 16 ==> RAG

    Q) Why RAG?
    Ans) 
        1. We cannot get the private data information
        2. Recent Data or Current affairs (Curently ChatGPT has access to the internet, but if you ask this to any open source models from Ollama or huggingFace then they will not be able to respond)
        3. Hallucination

        --> The solution to the above problem is - Fine-tuning

            Select any pre-trained LLM -> collect small Domain specific dataset with prompts and their desired output -> Choose the method of training(Full parameter, LoRA/QLoRA) -> Train for some epochs -> Evaluate & Safety test

        Different ways to provide Fine-tuning

            1. Supervised tuning (Labelled based)
            2. Unsupervised tuning (Continued pretraining - unlabelled)
            3. RLHF (Reinforcement Learning Human Feedback)

        Drawbacks of performing fine-tuning

            1. Training an huge LLM is computationally very expensive (solution to it is RAG, as in RAG we don't need to train the LLM we just need to update the vector store as per the knowledge base)
            2. Need to have Strong knowledge or technical expertise to train the model
            3. It is not feasible to train each once the knowledge gets updated

        --> The Other solution is "In-context Learning" (Learning through example given in the prompts (also called as few-shot learning))

            In-Context Learning is a core capability of Large Language Models (LLMs) like GPT-3/4, Claude, and Llama, where the model learns to solve a task purely by seeing examples in the prompt-without updating its weights.

        ----> The Advance or improved version of In-context learning is called as "RAG"
            RAG is a way to make a language model (like ChatGPT) smarter by giving it extra information at the time you ask your question, by providing the context in system message

        - *** There are mainly 4 steps while implementing RAG:

            1. Indexing - Creating the context to feed to the LLM from external knowledge base
            2. Retrivals - Retreiving the similar context from the knowledge base as asked by the user
            3. Augmentation - Creating the prompt(user prompt + context) for LLM to add-on the parametired knowledge of LLM with your extra knowledge
            4. Generation - Finally the response is generated by the LLM as per the prompts provided

        1. Indexing - It is the process of preparing your knowledge base so that it can be efficiently searched at query time. This steps consists of 4 sub-steps.

            1. Document Ingestion - Load your knowledge base into the memory (e.g PyPDFLoader, youtubeLoader, WebBaseLoader, gitLoader)
            2. Text chunking - Break large documents into small, semantically meaningful chunks (e.g RecursiveCharacterTextSplitter, SemanticChunker)
            3. Embedding Generation - Convert each chunk into a dense vector (embedding) that captures its meaning (e.g OpenAIEmbeddings)
            4. Storage in a Vector Store - Store the vectors along with the original chunk text + metadata in a vector database.  
            
                Vector DB options:  

                    - Local: FAISS, Chroma  
                    - Cloud: Pinecone, Weaviate, Milvus, Qdrant.

        2. Retrieval - Retrieval is the real-time process of finding the most relevant pieces of information from a pre-built index (created during indexing) based on the user's question.  
        
        It's like asking:  "From all the knowledge I have, which 3-5 chunks are most helpful to answer this query?"    

            It works in 3 steps

                i. Convert the query into the embeddings
                ii. Perform semantic search on each chunks stored in the vector store
                iii. Rank as per the match with the query (Closest to the query embedding)
                iv. Then finally retrieve the top matcher from the chunks

        3. Augmentation: It refers to the step where the retrieved documents (chunks of relevant context) are combined with the user's query to form a new, enriched prompt for the LLM.

        4. Generation: It is the final step where a Large Language Model (LLM) uses the user's query and the retrieved & augmented context to generate a response.
